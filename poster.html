Desktop Publication Management System  ·  C# · WPF · Entity Framework Core · MVVM
A Windows desktop application for managing the classic pubs database — enabling staff to manage sales orders, browse a book catalogue, and search publishers through a clean, modern WPF interface built on the MVVM pattern.
Project Objective

Build a fully functional multi-subsystem desktop application that demonstrates professional-grade software architecture, database integration, and user-interface design using the Microsoft .NET ecosystem.

Three subsystems delivered:

  • 🛒  Sales Order Management — full CRUD with validation
  • 📖  Book Search — catalogue browse & detail panel
  • 🏢  Publisher Search — publisher browse & detail panel
MVVM Architecture
┌──────────────────────────────────────────────────────────────┐ │ Views (XAML) │ │ MainWindow · SalesOrderView · BookSearchView · │ │ PublisherSearchView — zero code-behind │ └──────────────────────────────┬───────────────────────────────┘ │ Data Binding (INotifyPropertyChanged) ┌──────────────────────────────▼───────────────────────────────┐ │ ViewModels │ │ MainViewModel · SalesOrderViewModel · BookSearchViewModel │ │ PublisherSearchViewModel · BaseViewModel (SetProperty) │ └──────────────────────────────┬───────────────────────────────┘ │ EF Core Queries / RelayCommand ┌──────────────────────────────▼───────────────────────────────┐ │ Data Layer (EF Core + DbContext) │ │ PubsDbContext · DbContextFactory · appsettings.json │ └──────────────────────────────┬───────────────────────────────┘ │ ┌──────────────────────────────▼───────────────────────────────┐ │ SQL Server / LocalDB (pubs schema) │ │ publishers · titles · authors · stores · sales │ └──────────────────────────────────────────────────────────────┘

Workflow: User interacts with View → ViewModel executes RelayCommand → EF Core queries SQL Server → result bound back to View.

Design Decisions
Why MVVM?
Views are purely XAML with zero code-behind — any ViewModel can be tested independently of the UI, and swapping a View requires no logic changes.
Why a per-operation DbContext factory?
Avoids long-lived contexts; each DB call creates a fresh PubsDbContext to prevent stale-entity tracking and connection-pool exhaustion.
Why composite primary keys on Sales & TitleAuthor?
Mirrors the real-world pubs schema constraints, enforced at both the database and application level — duplicate detection runs before any insert.
Why RelayCommand?
Eliminates boilerplate ICommand implementations; any method — including canExecute predicates — is wrapped in one line and bound directly in XAML.
Why does MainViewModel own child ViewModels?
Provides centralized status-bar aggregation and single-window navigation without frames, pages, or navigation service overhead.
